## 结论

优先路线是: 先治理文档状态, 再完成 TUI 性能最小闭环, 随后把 Daemon 做成真实 Agent runtime. TUI 不应承担 1M 上下文全量渲染, 它只负责可见内容, 输入体验和事件展示. 全量内容应由 transcript, session store 和 context manager 管理.

## Phase A: 文档事实源治理

### 目标

修正设计文档和真实代码状态不一致的问题, 让后续开发基于可信事实推进.

### 任务

1. 修正 `docs/README.md` 中 Phase 3a/3b/3c 的完成状态描述.
2. 修正 `docs/architecture.md` 中前后矛盾的模块状态.
3. 将 Memory, MCP, Skills, Daemon 标注为真实状态:
   - Memory: SQLite 表结构存在, 但检索, session 生命周期和 FTS 写入闭环不足.
   - MCP: client/transport 存在, 但 ServerManager 到 ToolRegistry 的工具注册不完整.
   - Skills: 全局和项目加载存在, 但路径, frontmatter 和依赖能力未完全对齐设计.
   - Daemon: HTTP router 和 session id 管理存在, 但还不是 Agent runtime.
4. 建立或更新 `docs/10_事实源/` 中对应事实源文档.

### 验收

- 文档不再把骨架或部分实现标为完整实现.
- 每个模块都有唯一可信的当前状态说明.
- 新需求和阶段计划不再写进长期事实源正文.

## Phase B: TUI 性能最小闭环

### 目标

让 TUI 在长回复和工具输出场景下保持输入响应, 不因内容增长导致每帧渲染成本线性上升.

### 任务

1. 引入 dirty render:
   - 输入变化时重绘.
   - assistant delta 到达时重绘.
   - tool 状态变化时重绘.
   - resize 和命令面板状态变化时重绘.
   - 空闲时不做 30ms 无条件重绘.
2. 引入增量渲染缓存:
   - 对 `UiMessage` 或 transcript view 建立 cached lines.
   - 流式 delta 只追加解析新增内容.
   - 避免每帧对当前长消息完整调用 `message_lines_for`.
3. 限制 live 区渲染量:
   - 只渲染最后 N 行可见内容.
   - 完整内容写入 transcript 或终端 scrollback.
4. 大输出分块写入:
   - `insert_before` 每次写入限制行数.
   - 工具输出和长回复分块 flush.
5. 工具输出折叠:
   - 默认展示状态, 耗时, exit code, 前后若干行.
   - 完整输出写入文件或 transcript, UI 只给路径和摘要.
6. 添加性能基准:
   - 10KB, 100KB, 1MB, 10MB stream.
   - 记录单帧耗时, 输入延迟, 空闲 CPU 和内存增长.

### 验收

- 1MB assistant 输出时 TUI 不 panic, 输入仍可响应.
- 空闲状态 CPU 占用接近 0.
- 长工具输出不会直接撑爆 TUI 渲染缓冲.
- 用户仍可依赖终端默认 scrollback 查看历史展示内容.

## Phase C: Memory 闭环修正

### 目标

让 Memory 从“表结构和部分接口存在”变成可检索, 可追溯, 可接入 Agent 上下文的闭环能力.

### 任务

1. 修复 session 生命周期:
   - `session_start` 写入当前 session id, project, cwd.
   - `observe` 使用真实 session id, 不再固定 `global`.
   - `session_end` 更新结束状态.
2. 修复 FTS 写入:
   - observe/remember 同步写入 `obs_fts`.
   - 或使用 SQLite trigger 保持 FTS 一致.
3. 实现 `search` 闭环:
   - 返回真实 MemorySearchResult.
   - 接入 `build_context`.
4. 修复字符串截断:
   - 使用 char 边界截断中文标题.
5. 降低 async 阻塞:
   - 短期用 `spawn_blocking`.
   - 长期评估 dedicated DB worker.

### 验收

- observe -> FTS search -> build_context 有集成测试.
- 中文内容 remember 不 panic.
- 多 session 数据可以按 session/project 查询.

## Phase D: MCP 与 Skills 接线

### 目标

让 MCP 工具和 Skills 能稳定进入 Agent 主链路, 且行为和文档规格一致.

### 任务

1. MCP:
   - `McpServerManager::register_tools` 注册 `McpToolEntry`.
   - 工具名统一使用 `server/tool`.
   - MCP client 和工具元数据生命周期统一管理.
   - 增加 mock stdio server 集成测试.
2. Skills:
   - 决定项目级路径使用 `.qianxun/skills` 还是兼容 `.claude/skills`.
   - 支持目录 `README.md` skill.
   - 抽出共享 frontmatter parser.
   - 明确 `depends_on`, `priority`, `project_only` 是当前实现还是后续规划.

### 验收

- MCP list_tools 后工具能进入 ToolRegistry definitions.
- LLM 可以看到并调用 MCP 工具.
- Skills 文档示例能被真实 parser 解析.

## Phase E: Daemon 真实运行时

### 目标

把 Daemon 从 HTTP 骨架升级为唯一 Agent runtime, 让 TUI/ACP/Web 都成为 thin client.

### 任务

1. 重构 `AgentLoopHost`:
   - 持有真实 session runtime.
   - 每个 session 包含 conversation, config, provider handle, tool registry, memory context.
2. 实现 prompt SSE:
   - 调用 `processing_loop::handle_user_message`.
   - 输出 assistant delta, tool_start, tool_result, usage, done, error 事件.
3. 实现 session store:
   - conversation 持久化.
   - transcript/event log 持久化.
   - 支持恢复, 列表, 删除.
4. TUI/ACP 改为 Daemon client:
   - 独立 standalone 模式保留为开发调试入口.
   - 默认连接本地 Daemon.

### 验收

- `qx daemon` 可创建 session 并真实调用模型.
- TUI 和 ACP 复用同一个 Daemon session/runtime.
- Daemon 重启后可以恢复历史 session.

## Phase F: Agent Patterns 与工具安全策略

### 目标

把 Agent Patterns 从类型和模块骨架接入真实 AgentLoop, 并补齐工具权限边界.

### 任务

1. Agent Patterns:
   - React 保持默认主链路.
   - Plan-and-Execute 接入两阶段执行.
   - Reflective 接入执行后自检.
   - Workflow 接入阶段模板.
2. Tool Policy:
   - 文件读写限制在 workspace 边界内.
   - 写文件, 执行命令, 网络访问按风险分类审批.
   - MCP 工具遵守 capabilities.
3. Context Policy:
   - tool result 统一截断和摘要.
   - 保证 tool_use/tool_result 配对.
   - 明确 token budget, compaction 和 memory recall 顺序.

### 验收

- 不同 Agent Pattern 有可运行端到端测试.
- 高风险工具调用会产生审批事件.
- workspace 外路径访问默认被拒绝或要求显式批准.

## Phase G: 清理旧入口与测试闸口

### 目标

降低维护成本, 建立后续迭代的回归保护.

### 任务

1. 清理旧 CLI REPL 残留:
   - 迁移仍有价值的 `/sessions`, resume, delete 等能力到新 TUI 或 Daemon.
   - 删除或归档不可达代码.
2. 建立测试闸口:
   - `cargo test --workspace`.
   - Memory 集成测试.
   - MCP mock server 测试.
   - Daemon prompt SSE 测试.
   - TUI render perf microbench.
   - Tool policy sandbox/approval 测试.
3. 建立验收文档:
   - 性能数据进入 `docs/40_验收/`.
   - 稳定设计进入 `docs/10_事实源/`.

### 验收

- warning 明显减少.
- 主要 Agent 主链路有端到端测试覆盖.
- 新功能修改必须能通过对应测试闸口.

## 执行优先级

| 优先级 | 阶段 | 原因 |
|---|---|---|
| P0 | Phase A | 先确保文档不误导开发 |
| P0 | Phase B | 直接改善当前 TUI 交互和性能 |
| P1 | Phase C | Memory 是长期上下文能力基础 |
| P1 | Phase D | MCP/Skills 是 Agent 工具扩展基础 |
| P1 | Phase E | Daemon 是更好 Agent 开发工具的架构核心 |
| P2 | Phase F | Agent Patterns 和安全策略提升可靠性 |
| P2 | Phase G | 降低维护成本, 固化质量门槛 |

## 当前建议

立即从 Phase B 开始实现 TUI 性能最小闭环, 同时并行做 Phase A 的文档状态修正. 这样可以最快改善用户可感知体验, 又不会让后续 Daemon-first 改造继续建立在不准确的文档之上.
