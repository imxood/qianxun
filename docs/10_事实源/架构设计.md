# 千寻 (Qianxun) 架构设计

> 状态: 初始落地 | 适用范围: Phase 1 代码框架 | 最后更新: 2026-05-26

## 1. 系统概述

千寻是一个 Rust 实现的个人 AI 系统，三种运行模式：

| 模式 | 说明 | 入口 |
|---|---|---|
| CLI REPL | 交互式终端，直接对话 | `qx` |
| ACP 协议 | 通过 stdio 与 Zed 编辑器通信 | `qx --acp-mode` (Phase 2) |
| Daemon | 后台服务 + HTTP API | 未来 Phase |

**设计哲学:**

- 分层解耦: core 不依赖任何 binary，OutputSink 让引擎不感知输出目标。
- 构建顺序交付: 4 个 Phase，每个交付可运行系统。
- 工具分三层: builtin / skill / MCP，统一通过 ToolRegistry 调度。

## 2. 核心类型 (`types.rs`)

**LlmError** — 统一的 LLM 错误类型:

```
NoApiKey, RateLimitExceeded, ApiError, AuthenticationError, PromptTooLarge, StreamEnded
```

**TokenUsage** — token 消耗追踪:

```rust
pub struct TokenUsage {
    pub input: u64,
    pub output: u64,
    pub cache_creation_input: Option<u64>,
    pub cache_read_input: Option<u64>,
}
```

**AgentConfig** — 代理引擎配置:

```rust
pub struct AgentConfig {
    pub max_turns: u32,
    pub max_retries: u32,
    pub max_tokens: Option<u64>,
    pub temperature: Option<f32>,
    pub thinking: ThinkingConfig,
}
```

其它类型: `StopReason` (EndTurn/MaxTokens/ToolUse 等), `ProviderCapabilities` (streaming/thinking/tool_use 能力声明), `ThinkingConfig` (Disabled/Enabled), `ToolChoice` (Auto/Any/Tool)。

## 3. Agent 引擎

### 3.1 消息模型 (`agent/message.rs`)

```rust
pub enum Message {
    User { id: UserMessageId, content: Vec<ContentBlock> },
    Assistant { id: AssistantMessageId, content: Vec<ContentBlock> },
}
pub enum ContentBlock {
    Text(String),
    ToolUse { id: ToolCallId, name: String, input: Value },
    ToolResult { id: ToolCallId, content: String, is_error: bool },
    Thinking { content: String, signature: Option<String> },
}
```

### 3.2 会话管理 (`agent/conversation.rs`)

`Conversation` 持有: system_prompt, messages, TokenBudget。关键方法:

- `build_request()` — 组装 `CompletionRequest`，注入 memory_context + skills_catalog 到 system prompt
- `enforce_budget()` — Phase 1 简单裁剪（按字符 1/4 估算 token，从最旧消息开始丢弃）
- `estimated_tokens()` — 估算当前消息列表的 token 消耗
- `push_user_message()` / `push_message()` — 添加消息并返回 ID

### 3.3 Agent 循环 (`agent/engine.rs`)

```
AgentState: Idle → WaitingLlm → ToolExecuting → (继续循环或停止)
AgentTransition: ContinueLlm / EndTurn / Retry / Error

AgentLoop:
  handle_user_message()
    ↓
  enforce_budget()
    ↓
  build_request() → stream_completion()
    ↓
  for each LlmStreamEvent:
   ├─ Text     → sink.on_text()
   ├─ ToolCall → ToolRegistry::execute()
   └─ Stop     → 有 tool calls 则继续，否则结束
```

### 3.4 系统提示词 (`agent/system_prompt.rs`)

`BASE_PROMPT` 常量定义千寻身份和核心原则。`build_system_prompt()` 组合 base prompt + 用户指令 + 上下文记忆 + 技能目录。

## 4. Provider 抽象 (`provider/`)

**LlmProvider trait:**

```rust
#[async_trait]
pub trait LlmProvider: Send + Sync {
    fn id(&self) -> &str;
    fn name(&self) -> &str;
    fn capabilities(&self) -> &ProviderCapabilities;
    async fn stream_completion(
        &self,
        request: CompletionRequest,
    ) -> Result<BoxStream<'static, Result<LlmStreamEvent, LlmError>>, LlmError>;
}
```

**CompletionRequest** — 请求结构: system, messages, tools, tool_choice, max_tokens, temperature, thinking, stop_sequences。

**LlmStreamEvent** — 流式事件枚举: Text(String), ToolCall { id, name, input }, Thinking { content, signature }, Stop(StopReason), Error(LlmError), TokenUsage(TokenUsage)。

Phase 1 为 trait 定义阶段，Provider 实现在 Phase 2。

## 5. 输出抽象 (`output.rs`)

**OutputSink trait** — 引擎不感知输出目标:

```rust
#[async_trait]
pub trait OutputSink: Send + Sync {
    async fn on_text(&self, text: &str);
    async fn on_thinking(&self, content: &str);
    async fn on_tool_call(&self, name: &str, args: &Value);
    async fn on_token_usage(&self, usage: &TokenUsage);
    async fn on_error(&self, error: &LlmError);
    async fn on_turn_finished(&self);
}
```

CLI 实现 (`CliOutputSink`): ANSI 终端输出，tool_call 以 `🔧 工具: name(args)` 格式显示。

## 6. 工具系统 (`tools/`)

**AgentTool trait:**

```rust
#[async_trait]
pub trait AgentTool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn input_schema(&self) -> Value;
    async fn execute(&self, arguments: Value) -> Result<ToolOutput, ToolError>;
}
```

**ToolRegistry** — 统一调度入口，持有 builtin 和 mcp_tools 两个 HashMap。`definitions()` 合并全部工具的 definition 供 LLM 使用。

Phase 1 内置工具（骨架）:

| 工具 | 说明 |
|---|---|
| ReadTextFileTool | 读取文件 |
| WriteTextFileTool | 写入文件 |
| SearchTool | 搜索文件 |

## 7. 上下文与记忆 (`context/`)

**ContextProvider trait:**

```rust
#[async_trait]
pub trait ContextProvider: Send + Sync {
    fn name(&self) -> &str;
    async fn query(&self, _query: &str) -> Vec<ContextDocument>;
}
pub struct ContextDocument {
    pub content: String,
    pub relevance: f64,
}
```

**MemoryManager** — Phase 1 骨架，实现 ContextProvider。后续迭代实现基于 TokenSum 的对话摘要记忆。

## 8. Skills 系统 (`skills/`)

**SkillManager** — Phase 1 骨架。`build_catalog_prompt()` 将所有已注册技能格式化为 System Prompt 可注入的目录文本。后续迭代实现从目录发现、按需加载 body。

## 9. MCP 系统 (`mcp/`)

**McpClient** — Phase 1 骨架。定义 `connect_stdio()` / `connect_http()` / `initialize()` / `list_tools()` / `call_tool()` 的签名，通过 `McpToolEntry` 将 MCP 工具注册到 `ToolRegistry`。

## 10. CLI 入口 (`qianxun-cli/`)

### 10.1 主入口 (`main.rs`)

clap 命令定义:

| 参数 | 说明 |
|---|---|
| `--acp-mode` | ACP 协议模式（Phase 2） |
| `-v / --verbose` | 调试日志 |
| `-m / --model` | 指定模型 |

二进制名称 `qx`。

### 10.2 REPL 循环 (`cli.rs`)

- 显示 `用户 > ` 和 `千寻 > ` 提示符
- 斜杠命令: `/quit`, `/help`, `/reset`, `/usage` (Phase 1 骨架)
- 普通消息通过 AgentLoop 处理并输出

### 10.3 CliOutputSink (`output.rs`)

ANS I 终端彩色输出实现，tool_call 使用 `🔧` 前缀，error 使用红色 `✖`。

## 11. 构建顺序

| Phase | 交付 | 核心任务 |
|---|---|---|
| 1 | 代码骨架 + 核心类型 + REPL CLI | 项目结构、类型定义、trait、空实现、REPL |
| 2 | ACP 协议 + Provider 实现 | ACP stdio server、至少一个 LLM Provider |
| 3 | Memory/Skills/MCP 集成 | 对话记忆、技能加载、MCP 工具调用 |
| 4 | Daemon 模式 + 完整 RAG | HTTP API、文件索引、知识库检索 |
