# 千寻 (Qianxun) 架构设计

> 状态: 生效 | 适用范围: Phase 1-2 | 最后更新: 2026-05-27 | 已评审

## 1. 系统概述

千寻是一个 Rust 实现的个人 AI 系统，三种运行模式：

| 模式 | 说明 | 入口 |
|---|---|---|
| CLI REPL | 交互式终端，直接对话 | `qx` |
| ACP 协议 | 通过 stdio JSON-RPC 2.0 与 Zed 编辑器通信 | `qx --acp-mode` |
| Daemon | 后台服务 + HTTP API | 未来 Phase |

**设计哲学:**

- 分层解耦: core 不依赖任何 binary，OutputSink 让引擎不感知输出目标。
- 构建顺序交付: 4 个 Phase，每个交付可运行系统。
- 工具分三层: builtin / skill / MCP，统一通过 ToolRegistry 调度。

## 2. 核心类型 (`types.rs`)

**LlmError** — 统一的 LLM 错误类型:
NoApiKey, RateLimitExceeded, ApiError, AuthenticationError, PromptTooLarge, StreamEnded

**TokenUsage** — token 消耗追踪 (input, output, cache_creation_input, cache_read_input)

**AgentConfig** — 代理引擎配置 (max_turns, max_retries, max_tokens, temperature, thinking)

其它类型: StopReason, ProviderCapabilities, ThinkingConfig, ToolChoice

## 3. 全局配置 (`config.rs`)

JSON 格式的全局配置文件（支持 `//` 注释，`~/.qianxun/config.json`）:

| 段 | 字段 | 说明 |
|---|---|---|
| deepseek | api_key, base_url, model | LLM Provider 配置 |
| agent | max_turns 等 | AgentLoop 配置 |
| budget | max_input_tokens, max_output_tokens | Token 预算 |

优先级链: CLI 参数 > 环境变量 `DEEPSEEK_API_KEY` > 配置文件 > 内置默认值。

`Config::from_file()` 读取 JSON 文件（支持 `//` 注释），`resolve()` 合并环境变量和参数覆写。

内置工具函数 `write_default_config()` 生成配置模板。

## 4. Agent 引擎

### 4.1 消息模型 (`agent/message.rs`)

```rust
pub enum Message {
    User { id, content: Vec<ContentBlock> },
    Assistant { id, content: Vec<ContentBlock> },
}
```

ContentBlock 类型: text, tool_use, tool_result, thinking

### 4.2 会话管理 (`agent/conversation.rs`)

`Conversation` 持有 system_prompt, messages, TokenBudget。

关键方法:
- `build_request()` — 组装 CompletionRequest，注入 memory/skills 到 system prompt
- `enforce_budget()` — 裁剪旧消息以控制 token 消耗
- `push_user_message()` / `push_message()` — 添加消息

### 4.3 Agent 循环 (`agent/engine.rs`)

```
AgentState: Idle → WaitingLlm → ToolExecuting → (继续循环或停止)

processing_loop::handle_user_message()
  → enforce_budget()
  → build_request() → provider.stream_completion()
  → 对每个 LlmStreamEvent:
     Text     → sink.on_text()
     Thinking → sink.on_thinking()
     ToolCall → sink.on_tool_call() + tools.execute() → push results → 继续循环
     Stop     → 有 tool_use 则继续，否则结束
```

### 4.4 系统提示词 (`agent/system_prompt.rs`)

`BASE_PROMPT` 定义千寻身份和核心原则。`build_system_prompt(workspace_context, skills_catalog, custom_instructions)` 组合各部分。工作区上下文在构建时注入，包含项目类型、根路径、CLAUDE.md 内容等。

## 5. Provider 抽象 (`provider/`)

**LlmProvider trait:**
```rust
pub trait LlmProvider: Send + Sync {
    fn id(&self) -> &str;
    fn capabilities(&self) -> &ProviderCapabilities;
    async fn stream_completion(&self, request: CompletionRequest)
        -> Result<BoxStream<Result<LlmStreamEvent, LlmError>>, LlmError>;
}
```

**实现: DeepSeekProvider** — 通过 DeepSeek Anthropic 兼容 API (`/anthropic/v1/messages`) 通信。使用 SSE 流式解析，支持 text/thinking/tool_use 事件和 token 用量跟踪。

## 6. 输出抽象 (`output.rs`)

**OutputSink trait** — 引擎不感知输出目标:

```rust
pub trait OutputSink: Send + Sync {
    async fn on_text(&self, text: &str);
    async fn on_thinking(&self, text: &str);
    async fn on_tool_call(&self, tool_call_id: &str, tool_name: &str, arguments: &Value);
    async fn on_token_usage(&self, usage: &TokenUsage);
    async fn on_error(&self, error: &LlmError);
    async fn on_turn_finished(&self, reason: &StopReason, usage: &TokenUsage);
}
```

两种实现:
- **CliOutputSink** — ANSI 终端彩色输出
- **AcpOutputSink** — 通过 mpsc 通道发送 session/update 通知

## 7. 工具系统 (`tools/`)

**AgentTool trait:**
```rust
pub trait AgentTool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn input_schema(&self) -> Value;
    async fn execute(&self, arguments: Value) -> Result<ToolOutput, ToolError>;
}
```

**ToolRegistry** — 统一调度入口，持有 builtin 和 mcp_tools 两个 HashMap。`definitions()` 合并全部工具的 definition 供 LLM 使用。`execute_async()` 通过真正的 async 执行工具；`execute()` 通过 `block_on` 驱动，用于非 tokio 上下文。

5 个内置工具:

| 工具 | 说明 |
|---|---|
| ReadTextFileTool | 读取文件（100KB 截断：保留前 50K + 后 50K） |
| WriteTextFileTool | 写入文件（自动创建父目录） |
| SearchTool | 递归 glob 文件名搜索（最多 200 条） |
| GrepTool | 文件内容搜索（子串匹配，行号，include glob 过滤，最多 200 条） |
| ListDirectoryTool | 目录树状列表（depth 1-3，显示文件/目录大小，隐藏点文件） |

在 ACP 模式下，ReadTextFileTool 和 WriteTextFileTool 被替换为转发版本 (ForwardingReadFileTool / ForwardingWriteFileTool)，优先通过 ACP bidirectional requests 请求编辑器代理执行，失败时回退到本地执行。

## 8. 工作区支持 (`workspace.rs`)

**ProjectType 枚举:**
```rust
pub enum ProjectType { Rust { is_workspace: bool }, Node, Python, Generic }
```

**Workspace 结构体:**
```rust
pub struct Workspace {
    pub root: PathBuf,
    pub project_type: ProjectType,
    pub project_instructions: Option<String>,
    pub detected_files: Vec<String>,
}
```

`detect_workspace()` 从指定目录向上查找（最多 3 层父目录），检测 Cargo.toml, package.json, CLAUDE.md, pyproject.toml 等项目标志文件。

`build_workspace_context()` 生成注入到系统提示词的文本块，包含项目根路径、类型、CLAUDE.md 内容、检测到的配置文件。

## 9. ACP 协议 (`qianxun-acp/`)

### 9.1 协议概述

ACP (Agent Communication Protocol) 是基于 JSON-RPC 2.0 的协议，通过 stdio 行分隔 JSON 通信。

**生命周期:**

```
Client → initialize         → Server (握手，声明能力)
Client → sessions/new       → Server (创建会话)
Client → session/prompt     → Server (流式执行)
Server → session/update     → Client (通知，推送流式结果)
Client → cancel             → Server (取消进行中的 prompt)
Client → sessions/close     → Server (关闭会话)
Agent  → fs/read_text_file  → Client (双向请求，请求编辑器读文件)
```

### 9.2 模块结构

| 模块 | 职责 |
|---|---|
| types.rs | JSON-RPC 2.0 信封 (Request/Response/Notification/Error) + ACP 协议类型 (Initialize, Sessions, Prompt, Update 等) |
| transport.rs | stdio 行帧读写；后台读取任务按类型路由消息；pending 请求 HashMap 用于双向请求关联 |
| session.rs | SessionManager 管理多会话生命周期 |
| acp_output.rs | AcpOutputSink 实现 OutputSink，通过 mpsc 通道发送 session/update 通知 |
| prompt.rs | session/prompt 桥接到 processing_loop 的占位模块 |
| handler.rs | 请求分发 (initialize/sessions/new/close/list/prompt/cancel) + ForwardingToolRegistry 构建 |
| server.rs | 主循环：select! 同时处理传入请求和输出事件 |

### 9.3 传输层设计

`AcpTransport` 内部使用:
- `Mutex<BufWriter<Stdout>>` — 写保护，多任务共享
- `Arc<Mutex<HashMap<RequestId, oneshot::Sender>>>` — pending 请求关联
- 后台 `tokio::spawn` 任务读取 stdin，解析 JSON，分类路由

双向请求流程:
1. Agent 工具调用 `transport.send_request("fs/read_text_file", params, timeout)`
2. 生成 ID，注册 oneshot sender
3. 写入 JSON-RPC 请求到 stdout
4. 等待 oneshot receiver（后台读取任务在收到对应 ID 的响应时触发 sender）
5. 返回响应结果或超时错误

## 10. 上下文与记忆 (`context/`)

**ContextProvider trait** 和 **MemoryManager** — 骨架状态，预留接口。

当前实现:
- `ContextDocument` 结构体和 `ContextProvider` trait 已定义
- `MemoryManager` 所有方法 (`read`/`write`/`build_context`) 返回空值
- 记忆上下文尚未注入 system prompt

后续规划: 基于文件系统的对话摘要记忆，跨会话保留关键上下文（Phase 3.1）。

## 11. Skills 系统 (`skills/`)

**SkillManager** — 骨架状态。`build_catalog_prompt()` 将所有已注册技能格式化为系统提示词可注入的目录文本。

当前实现:
- `Skill`/`SkillManager`/`SkillSource` 类型已定义
- `load_all()`/`read_body()`/`reload()` 均为空实现

后续规划: 从 `~/.qianxun/skills/` 和项目 `.claude/skills/` 加载技能 markdown（Phase 3.3）。

## 12. MCP 系统 (`mcp/`)

**McpClient** — 骨架状态。定义了 connect/initialize/list_tools/call_tool 签名。

当前实现:
- `McpServerConfig`/`McpClient`/`McpTool` 类型已定义
- `connect_stdio()`/`connect_http()`/`initialize()`/`list_tools()`/`call_tool()` 均为空实现
- `ToolRegistry` 中的 `mcp_tools` HashMap 可以 populate，但 `execute_async()` 尚未搜索 `mcp_tools`

后续规划: MCP 客户端实现 + ToolRegistry 接线（Phase 3.2）。

## 16. 当前架构分析

### 16.1 分层合理性评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 分层依赖 | ✅ | cli → acp → core，无循环依赖，core 无业务 crate 依赖 |
| 输出抽象 | ✅ | OutputSink trait 解耦引擎和输出目标，两种实现验证模式有效 |
| 工具系统 | ✅ | builtin/skill/MCP 三层设计，builtin 完整，后两层待接线 |
| Provider 抽象 | ✅ | LlmProvider trait 清晰，DeepSeek 完整实现 |
| Agent 引擎 | ✅ | Conversation 和 AgentLoop 职责分离，processing_loop 处理流式/工具/思考块 |
| 流式缓冲 | ✅ | 50 字节缓冲统一在 engine.rs 处理，避免各 sink 重复实现 |

### 16.2 已知问题

| 严重度 | 问题 | 位置 | 说明 |
|--------|------|------|------|
| 低 | `ProjectType::Python` 未赋值 | `workspace.rs` | 检测到 Python 文件仍归为 Generic |
| 低 | `AgentTransition` 未使用 | `engine.rs` | 枚举已定义但 processing_loop 不使用 |
| 低 | `TokenCounter` trait 未使用 | `conversation.rs` | 计数靠字符长度近似，trait 未接入 |
| 中 | MCP 工具执行未接线 | `tools/mod.rs` | `mcp_tools` 可 populate 但 `execute_async` 只查 `builtin` |
| 中 | Skills 空加载 | `skills/mod.rs` | `load_all()` 返回空 |
| 中 | 无持久化记忆 | `context/memory.rs` | 所有方法返回空 |
| 中 | 会话仅内存 | `qianxun-acp/session.rs` | 进程重启后丢失 |

### 16.3 设计取舍

- **Token 估计用字符近似**: 避免引入 tiktoken-rs 等 tokenizer 依赖，预算裁剪的精确性非关键路径
- **Session 只有内存存储**: Phase 2 目标是与 Zed 的工作会话，不需要跨进程持久化
- **`execute_async` 与 `execute` 共存**: `execute_async` 是真 async 调用，`execute` 是 block_on 包装用于非 tokio 上下文

### 16.4 Phase 3 规划

```
Phase 3.1: Memory 实现（最简单，推荐先做）
  → 基于文件的对话摘要记忆
  → 构建 memory context 注入 system prompt

Phase 3.2: MCP client 实现（中等复杂度）
  → 实现 connect_stdio/connect_http
  → 实现 initialize/list_tools/call_tool
  → 接线 ToolRegistry::execute_async 到 mcp_tools

Phase 3.3: Skills 实现（最复杂，推荐最后）
  → 定义 skill .md 格式规范
  → 从 ~/.qianxun/skills/ 和项目 .claude/skills/ 加载
  → 注入 system prompt
```

预计 Phase 3 完成后，千寻具备完整的上下文记忆、外部工具集成和技能加载能力，为 Phase 4（Daemon 模式 + RAG）奠定基础。
