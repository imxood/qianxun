# 14 缺口的可观测性规范

> 状态: 草稿 (待 code review) | 适用范围: 14 缺口运行的指标/日志/事件 | 最后更新: 2026-06-11 | 版本: v0.1

## 1. 目的

14 缺口叠加后, 系统复杂度上升。生产级必备: **出了问题能快速定位**。

3 层可观测性:
1. **tracing 日志** (Rust 进程内, 排查用)
2. **SseEvent** (前端可见, 用户感知)
3. **指标埋点** (后续 Prometheus 接入, 趋势监控)

## 2. tracing 日志规范

### 2.1 Span 命名约定

每个缺口的核心模块起一个 span:

```rust
// qianxun-core/src/hooks/registry.rs (缺口 01 + 09)
#[tracing::instrument(skip(self, event), fields(hook_name, tier))]
async fn dispatch(&self, tier: HookTier, event: HookEvent<'_>) -> HookResult {
    tracing::debug!(?tier, "hook dispatch start");
    // ...
}

// qianxun-core/src/provider/error.rs (缺口 02)
#[tracing::instrument]
fn classify(&self, status: Option<u16>, body: &str) -> LlmErrorKind {
    tracing::warn!(status, body_preview = %body.chars().take(200).collect::<String>(), "classify error");
    // ...
}

// qianxun-runtime/src/background_task.rs (缺口 05)
#[tracing::instrument(skip(self, spec), fields(task_id, parent))]
async fn start(&self, parent: SessionId, description: String, spec: SubAgentSpec) -> TaskId {
    tracing::info!(description, "background task started");
    // ...
}
```

### 2.2 日志级别使用

| 级别 | 用途 | 示例 |
|---|---|---|
| `ERROR` | 必须告警, 不可恢复 | LLM 全 provider 失败 / hook 熔断 / DB 写失败 |
| `WARN` | 可恢复但需关注 | 单 provider 失败 / verify Uncertain / flush 部分失败 |
| `INFO` | 关键状态变更 | task start/complete / session mode 切换 |
| `DEBUG` | 排查细节 | tool_call args / hook chain 选择 / recovery 决策 |

### 2.3 结构化字段

必带字段 (方便过滤):
- `session_id`
- `task_id` (如果有)
- `turn_id` (缺口 13 起)
- `provider` (provider 类错误)
- `gap_id` (如 "gap_02", 方便定位是哪个缺口的问题)

```rust
tracing::warn!(
    gap_id = "gap_02",
    session_id = %session_id,
    provider = "deepseek",
    status = 429,
    "rate limited, will retry"
);
```

## 3. SseEvent 规范 (前端可见)

### 3.1 已汇总的 14 缺口 SseEvent (见 16_接口契约汇总.md §4)

总新增 **21 个变体**:
- 缺口 01: 2 (HookDisabled, HookRecovered)
- 缺口 03: 1 (ToolDenied)
- 缺口 05: 5 (TaskQueued, TaskStarted, TaskProgress, TaskCompleted, TaskFailed)
- 缺口 13: 20 (AgentEvent) — 但**注意**: 20 个 AgentEvent 是统一的, 不是 20 个独立 SseEvent, 通过 `subscribe_events` 单独通道
- 缺口 11: 2 (VerifyStarted, VerifyCompleted)

### 3.2 前端 UI 映射 (SseEvent → 用户感知)

| SseEvent | Tauri UI 表现 |
|---|---|
| `HookDisabled` | 右下角 toast: "hook X 暂时禁用, 60s 后恢复" |
| `ToolDenied` | (缺口 03 专用) chat 区域显示 "sub-agent 越权: X" |
| `TaskQueued/Started/Progress/Completed/Failed` | col3 TaskListPanel 实时刷新 |
| `VerifyStarted` | chat 区域显示 "正在验证 spec X..." |
| `VerifyCompleted` | 显示 verdict + evidence 摘要 |
| `AgentEvent` (20 种) | 通过 `subscribe_events` 单独的 DevTools 面板 (开发时可见) |

### 3.3 SseEvent 命名约定

- 过去时态: `Started / Completed / Failed / Queued` (而非 Starting)
- 单数名词: `HookDisabled` (而非 HookDisable)
- 字段 snake_case + 字段顺序固定 (序列化稳定)

## 4. 指标埋点 (后续 Prometheus)

### 4.1 缺口对应核心指标

| 缺口 | 指标名 | 类型 | 用途 |
|---|---|---|---|
| 01 | `qianxun_hook_invocations_total{hook_name,result}` | counter | hook 调次数, 按 result 分桶 |
| 01 | `qianxun_hook_circuit_breaker_trips_total` | counter | 熔断次数 |
| 02 | `qianxun_llm_errors_total{kind}` | counter | LLM 错误按 kind 分桶 |
| 02 | `qianxun_llm_recovery_total{action}` | counter | 恢复动作分布 |
| 05 | `qianxun_background_tasks_active` | gauge | 当前 active 数 |
| 05 | `qianxun_background_task_duration_seconds` | histogram | 任务时长 |
| 06 | `qianxun_flush_triggers_total{trigger=soft/hard}` | counter | flush 触发按软/硬分 |
| 06 | `qianxun_flush_items_saved_total` | counter | 落盘 memory 条数 |
| 07 | `qianxun_dual_loop_inner_iterations_total` | counter | inner 步数 |
| 07 | `qianxun_steer_applied_total` | counter | steer 次数 |
| 08 | `qianxun_provider_layer_total{layer=1/2/3}` | counter | 各层调用次数 |
| 08 | `qianxun_provider_failover_total{from,to,reason}` | counter | 切换次数 |
| 08 | `qianxun_provider_circuit_open_seconds` | gauge | circuit open 时长 |
| 10 | `qianxun_stale_edit_total{tool}` | counter | stale edit 次数 (说明 LLM 行为问题) |
| 11 | `qianxun_verify_total{verdict}` | counter | verdict 分布 |
| 13 | `qianxun_knowledge_promote_total{from,to,gate_result}` | counter | promote 次数 + gate 拒绝数 |

### 4.2 暂时不接入 Prometheus

**现状**: 千寻没有 Prometheus exporter。

**策略**:
- 缺口实施时, **预留 `#[cfg(feature = "metrics")] counter!()` 调用点**
- 等专门工作项"指标系统"启动后再开 feature 接入
- 现阶段**主要靠 tracing 日志 + SseEvent 排查**

## 5. 排障手册 (按缺口)

| 缺口 | 常见问题 | 看哪个日志 | 看哪个 SseEvent |
|---|---|---|---|
| 01 | hook 反复失败 | tracing WARN "hook disabled" | HookDisabled |
| 02 | LLM 总 401 | tracing ERROR "auth permanent" | 无 (用户感知为 send_message 失败) |
| 05 | 任务卡 Queued | tracing DEBUG "queue size" | TaskStarted (没收到就是卡住) |
| 06 | flush 后 memory 没内容 | tracing DEBUG "flush items" | MemoryFlushed (AgentEvent) |
| 07 | steer 没生效 | tracing DEBUG "steer applied" | SteerApplied (AgentEvent) |
| 08 | provider 切换频繁 | tracing WARN "rotate provider" | 无 (用户感知为 send 慢) |
| 11 | verdict 永远 Uncertain | tracing WARN "verify uncertain reason" | VerifyCompleted |
| 13 | 知识不能 promote | tracing DEBUG "gate reject reason" | 无 |

## 6. 不在本设计范围

- 不实现 Prometheus exporter (后续工作项)
- 不做错误聚合 / 告警 / oncall
- 不做分布式 tracing (千寻单进程, 不需要)
- 不做用户行为分析
