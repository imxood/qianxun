# 千寻总设计

> 状态: 启动中 | 适用范围: 千寻整体架构 + 14 缺口整合 | 最后更新: 2026-06-11 | 版本: v0.1

## 1. 整体架构 (3 层)

```text
[基础设施层]  v2: HookRegistry + AgentMode + PermissionMode + SubAgent
                已有 10 个 crate 模块, ~2180 行
                详细: [agent_loop_v2.md](./agent_loop_v2.md)

[能力层]      14 缺口: 在 v2 之上叠加, ~4338 行
                入口: [00_总览.md](./00_总览.md)

[应用层]      Tauri Desktop + ACP stdio + TUI + WebSocket (未来)
                不在本设计范围
```

**本设计聚焦基础设施 + 能力层**: 怎么配合, 怎么避免重复, 怎么共享测试 / 配置 / 文档。

### 1.1 实施时序决策 (重要)

**问题**: v2 还在演进, 14 缺口跟 v2 阶段如何排序?

**选项分析**:

| 方案 | 描述 | 优 | 劣 |
|---|---|---|---|
| A. 先 v2 后 14 缺口 | v2 E1-E8 全做完, 再起 14 缺口 | v2 稳定, 14 缺口容易叠 | 用户等很久, ROI 慢 |
| B. 边 v2 边 14 缺口 | 每个 v2 阶段同时叠对应缺口 | 渐进交付 | 容易冲突, 测试散 |
| **C. 14 缺口独立 (本文档选)** | **14 缺口按依赖自驱动, 跟 v2 解耦** | **可立即启动, 灵活** | **部分缺口要等 v2 锚点 (E1/E2/E4)** |

**裁定**: 选 C, **14 缺口优先按自身依赖链启动**; 只有**显式依赖 v2 锚点**的缺口等锚点 (如缺口 13 双层循环等 v2 E2)。

**8 个无锚点可立即启动**: 02 LLM错误分类 / 04 Skill生命周期 / 06 Memory Flush / 07 Hook Tier / 08 Hashline Edit / 09 Context Window / 12 Provider Failover / 14 Knowledge Gate

**6 个有锚点**: 01 Hook 退出码 (E1) / 03 SubAgent 白名单 (E4) / 05 后台异步任务 (E4) / 10 Queue (E2) / 11 Verdict (E3) / 13 双层循环 (E2)

## 4. 14 缺口的 4 类分组 (按功能域)

按"共享代码 / 共享测试 / 同步发布"原则, 14 缺口分 4 组:

### A 组 — Hook 扩展 (2 项, ~255 行)

| 编号 | 主题 | 落点 |
|---|---|---|
| 01 | Hook 退出码 + 熔断 | `qianxun-core/src/hooks/registry.rs` |
| 07 | Hook 五层 Tier | `qianxun-core/src/hooks/tier.rs` (新) |

**共同点**: 都改 HookRegistry, 都引入新概念 (Error / Tier), 6 个 builtin hook 都要迁移。

**同步发布**: 2 个一起做, 一起测。

### B 组 — Provider 鲁棒性 (3 项, ~1135 行)

| 编号 | 主题 | 落点 |
|---|---|---|
| 02 | LLM 错误分类 | `qianxun-core/src/provider/error.rs` (新) |
| 12 | Provider 三层 Failover | `qianxun-core/src/provider/failover.rs` (新) |
| 09 | Context Window 5 层 | `qianxun-core/src/provider/capabilities.rs` (新) |

**依赖链**: 02 → 12, 09 独立可并行。

### C 组 — Memory 能力 (3 项, ~910 行)

| 编号 | 主题 | 落点 |
|---|---|---|
| 04 | Skill 生命周期 | `qianxun-core/src/skills/lifecycle.rs` (新) |
| 06 | Memory Flush | `qianxun-core/src/agent/context/compact.rs` (改) |
| 14 | Knowledge Gate | `qianxun-memory/src/knowledge.rs` (新) |

**依赖链**: 06 (Flush 落盘) → 14 (Gate 决定能否 promote), 04 独立。

### D 组 — 交互 / 编辑 / 验证 (6 项, ~2040 行)

| 编号 | 主题 | 落点 |
|---|---|---|
| 03 | SubAgent 工具白名单 | `qianxun-core/src/subagent/mod.rs` (改) |
| 05 | 后台异步任务 | `qianxun-runtime/src/background_task.rs` (新) |
| 13 | 双层循环 | `qianxun-core/src/processing_loop/dual.rs` (新) |
| 08 | Hashline Edit | `qianxun-core/src/tools/builtin/hashline.rs` (新) |
| 11 | Verdict + BDD | `qianxun-core/src/verify/mod.rs` (新) |
| 10 | Session Queue 5 mode | `qianxun-runtime/src/queue.rs` (新) |

**D 组最大风险**: 6 个缺口互相独立, 但都改 `processing_loop_v2`, 容易并发冲突。

**具体避免冲突方案**:
1. **逐缺口 PR**: 一个缺口 = 一个 PR, 不批量
2. **接口锁定优先**: 先做 D 组公共接口 (`processing_loop/v2.rs` 抽出 trait), 后做实现
3. **顺序**: 03 → 05 (都等 v2 E4) → 13 (等 v2 E2) → 11 (跟 reflect) → 08 → 10
4. **测试隔离**: 每个缺口的集成测试用**独立 session_id**
5. **冲突解决**: 冲突时按缺口编号小的优先
