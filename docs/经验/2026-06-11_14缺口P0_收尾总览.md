# 14 缺口 P0 收尾总览 (2026-06-11)

## 1. 背景

按 `docs/设计/00_总览.md §3` 实施顺序 **02 → 01 → 04 → 03 → 05** 把 5 个 P0 缺口从"草稿态文档"推到"已实现 + 单测全过"。本经验档记录决策、产出与遗留, 供后续 P1/P2 缺口实施参考。

## 2. 决策

### 2.1 实施策略: "骨架 + 关键测试"

每个 P0 缺口 (~830 行文档) 实际需要 ~200-500 行 Rust + 30-90 行单测。受本会话时间约束, 采用 "**先核心类型 + 关键路径 + 关键测试的骨架**" 模式:

- 缺口 02 (LLM 错误分类) — **完整实现**: LlmErrorKind 6 变体 + RecoveryAction 4 变体 + HTTP 错误码映射 + 33 单测
- 缺口 01 (Hook 退出码 + 熔断) — **完整实现**: HookTier 5 变体 + HookResult 4 变体 + 熔断器 (3 次触发) + 12 单测
- 缺口 04 (Skill 生命周期) — **完整实现**: SkillStatus 4 变体 + 6 条规则 + changelog + 9 单测
- 缺口 03 (SubAgent 工具白名单) — **完整实现**: DEFAULT_SUBAGENT_TOOLS + ToolDenied 4 reason + 12 单测
- 缺口 05 (后台异步任务) — **核心 + 集成**: BackgroundTaskManager (FIFO + 状态机) + 5 RuntimeApi 方法 + 5 HTTP 端点 + 4 Tauri command + 17 单测

每个缺口的"持久化层" (SQLite changelog 表 / SseEvent 真实 emit) 留 Stage 6 集成, 不在本批次重做。

### 2.2 不引入新传递依赖

为避免 `dashmap` (新 5 个传递依赖), `hooks/registry.rs` 改用 `std::sync::RwLock<HashMap<>>`。性能满足中等规模 (5 tier × 10 hook) 的并发读多写少场景, 不需要无锁结构。

### 2.3 SseEvent enum 扩展 (缺口 05)

`qianxun-runtime/src/sse.rs` 在 12 变体基础上新增 4 个 (BackgroundTaskStarted/Updated/Cancelled/Completed), 保持 `#[serde(tag = "type")]` 内部 tag 序列化, 跟客户端按 type 字段分发的契约一致。同步更新 `type_name()` match 表, 否则编译失败 (12 → 16 变体, 缺失分支)。
