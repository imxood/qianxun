# ADR-0001: 记忆子系统数据库选型

> 状态: 已采纳 | 适用范围: `qianxun-memory` crate | 最后更新: 2026-05-31

## 背景

千寻记忆子系统（`qianxun-memory`）需要一个嵌入式数据库来存储 Observation、Memory、Session 等结构化数据，同时需要支持全文搜索和向量检索。

原始设计选型为 **redb**（纯 Rust KV 嵌入式数据库），通过以下方式实现搜索：
- 结构化查询：全表扫描 + 反序列化 → 内存过滤
- 全文搜索：自建 BM25 HashMap（5 秒防抖全量序列化到 redb）
- 向量搜索：内存 HashMap + WAL 追加持久化

## 可选方案

| 方案 | 类型 | 语言 | 说明 |
|---|---|---|---|
| **redb**（当前） | KV | Rust | 纯 Rust，但结构化查询需全表扫描 |
| **sled** | KV | Rust | 同 redb 的 KV 限制，项目维护停滞 |
| **SQLite** | 关系型 | C | 成熟度极高，FTS5 内置全文搜索 |
| **DuckDB** | 列式 OLAP | C++ | 嵌入式列存，点查慢 |
| **libsql** | 关系型 | C | SQLite fork + 异步原生，pre-1.0 |

## 数据访问模式分析

判断标准：**memory 子系统的核心操作是 OLTP 还是 OLAP？**

| 操作 | 频率 | 模式 | SQL 表达 |
|---|---|---|---|
| `observe()` 写入 1 条 Observation | 1-10/s | 单行 INSERT | `INSERT INTO observations VALUES (...)` |
| `build_context()` 搜索 | 每轮 1 次 | 点查 + FTS 搜索 | `SELECT ... FROM obs_fts WHERE MATCH ?` |
| `consolidation()` 按 session 读 | 每次 session 结束 | 小范围扫描 | `SELECT ... WHERE session_id = ?` |
| `search()` 按 query 查 | 用户主动触发 | FTS 搜索 | `SELECT ... FROM obs_fts WHERE MATCH ?` |

**结论：OLTP 模式**。所有操作都是点查或小范围索引扫描，不需要 OLAP 的聚合分析能力。

## 决策

**选择 SQLite (rusqlite bundled)**。

### 理由

1. **结构化查询**：SQLite 索引查询 O(log n)，redb 全表扫描 O(n)。Observation 到 10 万条时差距显著。
2. **全文搜索**：SQLite FTS5 内置 BM25 搜索，可持久化。消除自建 BM25 HashMap + 全量序列化的问题。
3. **Schema 演进**：SQLite `ALTER TABLE` 版本化迁移。redb JSON blob 改字段名即炸旧数据。
4. **调试能力**：`sqlite3 mem.db` 即可查，开发效率提升显著。
5. **向量存储**：BLOB 列替代 WAL 追加 + 启动重建，每行 ~1.5KB 自动管理。
6. **稳定性**：35 年生产验证，每天运行在 ~1 万亿设备上。

### 代价

| 项 | 影响 |
|---|---|
| 构建时间 | 首次编译 SQLite bundled +60-120s |
| 二进制体积 | +~1MB（stripped, bundled） |
| 纯 Rust | 放弃（引入 C 编译依赖） |
| 异步支持 | 仍需 spawn_blocking（rusqlite 非原生 async） |

### 绑定方式

- **crate**: `rusqlite = "0.34"`
- **features**: `bundled`（自动编译 sqlite3.c）+ `vtab`（FTS5）+ `column_decltype`
- **C 工具链**: Windows 上使用 MSVC 或 MinGW（预装），无需额外安装

## 被排除的方案

### sled

- 纯 Rust ✅
- KV 模式同 redb，不支持结构化查询
- 项目维护基本停滞（v1 未发布，已知 bug 未修复）
- ❌ 不选

### DuckDB

- SQL 兼容 ✅
- 列式存储对聚合分析好，但千寻 memory 操作全是 OLTP 点查
- 点查延迟 ~100μs（SQLite ~5μs），最小内存 ~30-50MB
- 编译时间 ~5-10 分钟
- ❌ 不选

### libsql

- 本地嵌入 ✅，原生 async ✅
- 本地嵌入 + 远程复制 —— 多机同步的最佳候选
- pre-1.0 API 有变动风险
- 生态系统远不如 SQLite（工具链、调试、文档）
- ⚠️ 保留评估，可在 SQLite 基础上抽象一层以降低切换成本

## 后续影响

- `memory-design.md` 存储层全面重写（已同步更新 v0.2）
- 表定义从 redb 的 8 张 TableDefinition 改为 SQLite DDL
