用户问:
> "还需要考虑的一点是 plan模式 与 `bypass permissions on`模式, 是否输出 这些模式里面的? 当前的代码架构 或许已经不是最优的了, 如果可以的话 请重新设计"

- `PlanState::WaitingApproval` + `UserDecision` 是死代码 (`qianxun-core/src/agent/plan.rs:33,40`)

决策: 重写 v2 文档, 加双轴模式 + 2 个新 hook (PlanGateHook / PermissionGateHook)。**v2 文档从 ~13723 字符扩到 ~21427 字符**。

## 实施风险记录

### 14 缺口分散 → 接口不统一

**风险**: 14 个独立 PR 可能产生不一致的接口风格
**缓解**: 每个缺口落 `qianxun-core` 接口层, 不直接耦合到 Tauri/desktop

### Skill 自学习 (缺口 4) 复杂度高

**风险**: skill 自动蒸馏 + changelog + 归档, 涉及 LLM 评估, 可能效果差
**缓解**: **后置, 不阻塞 P0**, 跟其他 P0 解耦

### 双层循环 (缺口 7) 重构 v2 现有

**风险**: outer/inner 拆分会改 v2 E2
**缓解**: v2 E2 完成后启动, v1 保留不变, 走 opt-in 模式

### Provider Failover (缺口 8) 投资大

**风险**: 510 行 + 评分系统 + 熔断, 单 PR 难 review
**缓解**: 拆 3 个 sub-PR (Retry → Chain → Router), 排最后

## 重要判断

1. **14 缺口是叠加不是替代**: v2 基础设施不动, 14 缺口**在 v2 之上加能力**
2. **借鉴源不完整移植**: 只取所需, 不整套搬 (跟 openfang 整体架构不同)
3. **借鉴源标清楚**: 每个 commit message 标借鉴源, 方便后续 review
4. **单 dev 1-2 月工作量**: ~4340 行, 每天 100-150 行可完成
5. **不引入新外部依赖**: 14 缺口用 Rust 标准库 + 现有 crate
